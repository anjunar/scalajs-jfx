jfx.layout.Viewport
Viewport
A shared layer for windows, notifications and overlays. Viewport lets the app keep richer interaction layers inside the same runtime and visual shell.
Operational Fit
Supports desktop-style windows without leaving the page app.
Handles overlays and toast-like notifications from the same API family.
Useful for docs, admin tools, editors and inspection workflows.
API Snapshot
viewport { ... }
Mount the shared viewport layer in the shell.
WindowConf
Describe floating windows declaratively.
captureComponent
Reuse DSL content inside a viewport-managed window.
notify(...)
Show lightweight notifications.
Import Surface
import jfx.layout.Viewport import jfx.layout.Viewport.{WindowConf, viewport} import jfx.action.Button.button
Usage Skeleton
viewport { mount(inject[Router]) } button("Open Inspector") { onClick { _ => Viewport.addWindow( WindowConf( title = "Profile Editor", width = 480, height = 280, component = Viewport.captureComponent { ... } ) ) } }
Integration Patterns
App-Level Host
Mount the viewport once near the shell so every page can open notifications and windows.
Floating Tools
Use windows for secondary workflows like inspectors, editors or previews.
Reused DSL Content
Capture normal DSL components into a window instead of creating a separate rendering path.
Live Sandbox
One shared layer
Notifications and windows come from the same runtime surface instead of ad hoc DOM overlays.
Composable content
The example window is built from the same DSL components as the page itself.
Low-friction tooling
This is a strong fit for inspectors, editors, previews and docs-side utilities.
Open runtime layers without leaving the docs page.
viewportActions=notification | sampleWindow
Forms, tables, windows and docs all run on the same component model.
